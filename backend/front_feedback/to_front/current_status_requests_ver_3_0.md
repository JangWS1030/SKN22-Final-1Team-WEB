# 프론트 후속 확인 요청 `ver_3_0`

이 문서는 아래 두 경우를 함께 고려해서 작성했습니다.
- 프론트 개발자 본인이 바로 읽고 이해하기 쉬운 내용
- GPT 에이전트가 옆에서 도와줄 때 그대로 참고하기 쉬운 내용

같은 이슈를 PM에게도 공유하지만, 이 문서의 초점은 다릅니다.
- 백엔드가 어디까지 준비했는지
- 프론트에서 무엇을 정하거나 확인해야 하는지
- 그 답이 화면 구현에서 왜 중요한지

## `#32` 매장 계정 생성

백엔드가 이미 준비한 것:
- 관리자/매장 계정 생성 API가 있습니다.
- 동의 항목 저장과 DB 저장 흐름이 준비되어 있습니다.
- 가입 성공 응답 자체는 줄 수 있습니다.

프론트에서 확인이 필요한 것:
- [ ] 가입 성공 후 첫 화면을 어디로 보낼지 확정
- [ ] `partner/signup/`를 실제 가입 화면으로 사용할지 확정
- [ ] 가입 성공 후 자동 로그인인지, 로그인 화면 이동인지 확정

왜 필요한가:
- 계정 생성 자체는 백엔드가 처리하지만, 가입 직후의 사용자 흐름은 프론트 화면 기준이 먼저 정해져야 합니다.

## `#123` 추천 시뮬레이션 재생성 API 제공

백엔드가 이미 준비한 것:
- 공개 재생성 endpoint가 있습니다.
- `recommendation_id` 또는 snapshot 기반으로 재생성 요청을 받을 수 있습니다.
- 재생성 성공 시 카드 형태의 응답을 줄 수 있습니다.

프론트에서 확인이 필요한 것:
- [ ] 재생성 버튼을 어느 화면에 둘지 확정
- [ ] `recommendation_id`를 프론트 상태에서 어떻게 보관할지 확정
- [ ] 대기 / 실패 / 완료 상태를 어떤 UI로 보여줄지 확정
- [ ] 카드 갱신을 전체 재조회로 할지, 부분 갱신으로 할지 확정

왜 필요한가:
- 백엔드는 재생성할 수 있지만, 사용자가 언제 어떻게 누르고 결과를 어떻게 보게 할지는 프론트가 정해야 합니다.

## `#140` 공통 에러 응답 표준화

백엔드가 이미 준비한 것:
- 현재 에러 응답 기준선을 helper로 모아둘 준비는 되어 있습니다.
- 지금 기본값은 DRF 스타일 `detail` 응답입니다.

프론트에서 확인이 필요한 것:
- [ ] 당장은 DRF 기본 에러 형식으로 먼저 붙여도 되는지 확인
- [ ] 최종적으로 어떤 전역 에러 형태를 원하는지 정리
- [ ] UI에서 실제로 필요한 필드가 `error_code`, `message`, `detail` 중 무엇인지 정리

왜 필요한가:
- 지금 프론트가 다른 에러 구조를 가정하고 구현하면, 나중에 에러 처리 코드를 다시 고쳐야 할 수 있습니다.

## `#141` 관리자 및 클라이언트 인증 갱신 정책 정리

백엔드가 이미 준비한 것:
- 현재는 access token 중심 구조입니다.
- `Bearer` 토큰 기준으로 동작하고, refresh token은 아직 없습니다.

프론트에서 확인이 필요한 것:
- [ ] 토큰 저장 위치 확정
- [ ] 만료 시 UX: 재로그인 / silent refresh / 자동 로그아웃 중 어떤 방식인지 확정
- [ ] refresh token 없이 먼저 연동 가능한지 확인

왜 필요한가:
- 백엔드는 현재 정책을 유지할 수 있지만, 실제 세션 경험은 프론트 UX 결정에 달려 있습니다.

## `#135` 모델 entrypoint, endpoint 추가

백엔드가 이미 준비한 것:
- 백엔드에서 모델 쪽으로 연결하는 경로는 있습니다.
- 현재 이슈는 연결 자체보다, 배포된 RunPod 동작과 문서 계약이 완전히 맞는지에 가깝습니다.

프론트에서 확인이 필요한 것:
- [ ] 모델 호출이 대기 / 실패 상태일 때 화면에서 어떻게 보여줄지 확정
- [ ] 모델 상태가 불안정할 때 재시도 또는 안내 문구를 어떻게 처리할지 정리

왜 필요한가:
- 백엔드는 모델 호출 경로를 유지할 수 있지만, 사용자에게 어떤 상태를 어떻게 보여줄지는 프론트 기준이 먼저 필요합니다.
